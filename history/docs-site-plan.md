# DS Kit Docs Site Plan

Date: 2026-04-09

## Decision

Build a small custom `Next.js` docs app modeled on `agent-browser` docs.

Keep it intentionally narrow:

- one docs layout
- MDX route pages
- one hand-authored nav config
- one plain `/kitchen-sink` route
- one `/install` route for copy-paste install paths
- optional `/registry` route if registry install and copy-first install need to be separated

Do not start with a docs framework.

Do not start with:

- AI docs chat
- custom search
- complex content indexing
- CMS
- autogenerated nav
- heavy design system marketing pages

## Why this shape

This repo is small and canonical. The docs should explain the kit, not become another product.

The `agent-browser` docs are a good reference because they are:

- custom but not overengineered
- MDX-first
- visually intentional
- easy to understand from the file tree

For `ds-kit`, this shape is a better fit than Mintlify or Fumadocs because the repo already wants local ownership and sharp boundaries.

## Product shape

Two jobs, clearly separated:

1. Docs

Explain:

- what `ds-kit` is
- what is included
- how install paths work
- how the token and component contract works
- what should stay local vs be promoted back here

2. Kitchen sink

Show the real primitives and states in one place:

- typography
- buttons
- field states
- loading states
- semantic colors
- motion samples if needed

The kitchen sink is not prose docs. It is a live inspection surface.

## Route map

Minimum useful routes:

- `/`
  - short introduction
  - what ships in V1
  - links to install and kitchen sink
- `/install`
  - copy-paste install paths
  - `shadcn` registry path
  - copy-first/manual path
  - dependencies and assumptions
- `/contract`
  - token system
  - theme bridge
  - merge rules
  - primitive ownership
- `/components`
  - what primitives exist
  - what is intentionally not included
- `/kitchen-sink`
  - live visual test page
- `/registry`
  - optional
  - only keep this if registry install deserves its own page distinct from `/install`

Nice-to-have later:

- `/promotion-rule`
- `/changelog`

## What to copy from Agent Browser

- app-router + MDX route structure
- one root layout with sidebar + mobile nav
- one tiny manual nav config
- simple prose styling
- restrained visual tone

Files to conceptually mirror:

- `docs/src/app/layout.tsx`
- `docs/src/lib/docs-navigation.ts`
- `docs/src/components/docs-sidebar.tsx`
- `docs/src/app/**/*.mdx`

## What not to copy from Agent Browser

Skip these on V1:

- `api/search`
- `api/docs-chat`
- `docs-chat.tsx`
- markdown-to-search indexing
- resizable AI panel
- provider-specific product pages
- feature sprawl

Those are good product-specific additions there, but unnecessary maintenance here.

## Kitchen sink scope

The kitchen sink should be brutally practical.

Sections:

- Typography
  - all `Text` sizes, weights, semantics
- Buttons
  - every variant
  - every size
  - loading
  - disabled
- Fields and states
  - focus
  - invalid
  - disabled
- Tokens
  - semantic surfaces
  - borders
  - foreground pairs
- Motion
  - only if motion tokens/animations are part of the shipped contract

Kitchen sink rules:

- use the real components from this repo
- no fake demo-only styling system
- every showcased state should correspond to shipped code

## Install page scope

`/install` should carry two install paths clearly:

1. Registry install

- `npx shadcn@latest add @ds-kit/core`
- registry config snippet

2. Copy-first/manual install

- exact files involved
- dependency assumptions
- global CSS entrypoint notes

It should be obvious which path is recommended and when.

## Design direction

Use the Agent Browser docs as a tone reference, not a clone target.

Desired qualities:

- compact
- crisp
- calm
- not startup-marketing
- not generic docs-template slop

For `ds-kit`, the docs should feel slightly more editorial and design-system-native:

- tighter type hierarchy
- more visible relationship to tokens and primitives
- better live examples

## Tech choices

Recommended:

- `Next.js`
- App Router
- MDX in route files
- Tailwind v4
- reuse `styles/` tokens where it makes sense

Avoid early:

- contentlayer
- CMS
- Algolia
- SaaS docs hosting lock-in

## Open implementation question

There are two clean ways to host this:

1. Add a self-contained `docs/` app inside this repo
2. Turn the repo root into a docs app and keep source files alongside it

Recommendation:

Use `docs/` as a self-contained app.

Why:

- keeps canonical source files separate from the docs app
- keeps package/tooling boundaries cleaner
- matches the `agent-browser` reference shape
- easier to delete, rebuild, or move later

## Build order

1. Create `docs/` Next app shell
2. Add shared layout and nav
3. Add `/`, `/install`, `/contract`, `/components`
4. Add `/kitchen-sink`
5. Reuse repo source components in live examples
6. Decide whether `/registry` should remain separate or fold into `/install`
7. Add polish only after the IA is proven

## Success criteria

The docs are good enough when:

- a new repo can install `ds-kit` without reading source files first
- the contract is understandable without reading three separate markdown files
- the kitchen sink becomes the fastest way to visually verify primitives
- the site remains small enough that maintaining it feels obvious, not annoying
